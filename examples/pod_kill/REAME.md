# Kubernetes Resiliency Experiment Analysis

## Context

You are analyzing the results of a **Kubernetes resiliency experiment**.

The experiment intentionally terminates pods in a workload to verify whether the system:

- remains **available**
- **recovers automatically**
- maintains **acceptable latency and error rates**

The platform executes controlled experiments using the following lifecycle:

```

Guardrails → Action → Rollback

```

- **Guardrails** ensure the system is safe to disrupt.
- **Action** injects the failure (pod termination).
- **Rollback** waits until the system recovers (pods respawn).

An **observer continuously probes the workload endpoint** to measure:

- HTTP latency
- HTTP errors
- availability

The observer runs in three phases:

```

Baseline (warmup) → Disruption → Recovery

````

Synthetic probe traffic is generated by the observer.
Note: **The workload may already be receiving real user traffic in addition to these probes.**

# Evaluation Criteria

Determine whether the system behaved correctly during the experiment.

Consider:

* Was the workload healthy during the **baseline period**?
* Did the disruption cause **errors or downtime**?
* Did latency **spike significantly** during disruption?
* Did the system **recover and restore the desired replica count**?
